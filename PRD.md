# Github-Projects
New feature for Github enabling students to explore &amp; learn skills by developing hands-on projects

---

# 📄 Product Requirements Document (PRD)

## Product: **GitHub Learn (Working Title)**

Built on: GitHub

---

# 1. 🧠 Problem Statement

Fresh graduates entering the software industry lack **practical, real-world development skills** despite strong academic performance.

Current ecosystem gaps:

* Platforms like LeetCode focus on algorithmic problem solving, not real-world development.
* Course platforms like Coursera optimize for content consumption, not skill mastery.
* AI tools like GitHub Copilot accelerate coding but don’t build foundational understanding for beginners.

👉 Result:

> Students struggle with building, debugging, collaborating, and shipping real software.

---

# 2. 🎯 Objective

Transform GitHub into a **learning-by-building platform** that helps students:

* Discover meaningful projects
* Build real-world applications
* Receive actionable feedback
* Showcase proof of work
* Transition seamlessly into industry-ready developers

---

# 3. 👤 Target Customer (ICP)

### Primary Segment:

* CS/IT students (2nd–4th year)
* Basic coding knowledge (DSA, APIs, Git)
* Struggling with “what to build” and “how to improve”

### Secondary Segment:

* Self-taught developers
* Early-career engineers (0–2 years)

---

# 4. 💡 Product Vision

> “From learning concepts → to shipping real-world software → to getting hired”

GitHub evolves from:

* Code hosting → **Developer growth platform**

---

# 5. 🧩 Key Features

---

## 5.1 🎯 Project Recommendation Engine

### Description:

Personalized project suggestions based on:

* Skill level
* Tech stack
* Interests

### Output:

* Structured “Project Paths”

### Example:

* Build a URL Shortener (Backend)
* Build a Real-time Chat App (Full-stack)

---

## 5.2 🧪 Guided Project Mode

### Description:

Step-by-step real-world project execution framework.

### Components:

* Task-based workflow (like Jira tickets)
* Context-driven instructions
* Hints instead of solutions

### Goal:

Simulate **real software development lifecycle**

---

## 5.3 🤖 AI Learning Assistant

Powered by:

* GitHub Copilot

### Capabilities:

* Context-aware hints
* Code explanation
* Bug identification
* Concept clarification

### Design Principle:

> Guide, don’t solve

---

## 5.4 🧑‍💻 Feedback System

### 3 Layers:

#### 1. AI Code Review

* Code quality
* Performance suggestions
* Best practices

#### 2. Peer Review

* PR-style comments
* Upvotes for quality solutions

#### 3. Mentor Review (Future)

* Industry experts

---

## 5.5 🌍 Social & Community Layer

### Features:

* Project feed (discover builds)
* Dev logs (“building in public”)
* Idea boards (problem-solving discussions)

Inspired by:

* LinkedIn
* Reddit

---

## 5.6 🏆 Gamification System

### Philosophy:

Reward **outcomes, not activity**

### Examples:

* “First Feature Shipped”
* “Bug Resolved”
* “Optimized Performance”

---

## 5.7 📊 Skill Graph (Portfolio Layer)

### Replaces:

* Resume-based claims

### Shows:

* Projects built
* Code quality
* Collaboration history
* Problem-solving ability

---

## 5.8 🏢 Recruiter Discovery Layer (Future Phase)

### Enables:

* Companies to evaluate:

  * Real code
  * Thinking patterns
  * Collaboration

---

# 6. 🔁 User Journey

1. User logs into GitHub
2. Sees “Start Building”
3. Gets project recommendations
4. Starts Guided Project
5. Uses AI assistant when stuck
6. Pushes code → gets feedback
7. Shares progress in community
8. Completes project
9. Builds portfolio automatically
10. Gains visibility to recruiters

---

# 7. 📏 Success Metrics

---

## 7.1 🎯 North Star Metric

> **Number of Projects Successfully Completed per Active User**

Why:

* Measures real learning
* Indicates depth, not vanity engagement

---

## 7.2 📊 Core Metrics

### Acquisition

* % of student users activating “Learn Mode”
* Project start rate

### Engagement

* Weekly Active Learners (WAL)
* Avg. time spent per project
* Tasks completed per project

### Retention

* Day 7 / Day 30 retention
* % users starting second project

### Learning Effectiveness

* Project completion rate
* Reduction in AI dependency over time
* Improvement in code quality scores

### Community

* % users contributing feedback
* PR comments per project
* Content (posts/dev logs) creation rate

### Conversion (Future)

* Recruiter views per profile
* Shortlist rate

---

# 8. 🎯 Success Criteria

---

## Phase 1 (0–6 months): MVP Validation

* ≥ 30% of new student users start a project
* ≥ 40% project completion rate (guided projects)
* ≥ 25% Day-7 retention
* ≥ 2 feedback interactions per user

👉 Signal: Product solves real learning gap

---

## Phase 2 (6–12 months): Engagement & Habit Formation

* ≥ 50% users start second project
* ≥ 35% Day-30 retention
* ≥ 3 projects per active user (avg.)
* Strong community participation (>40% users engage socially)

👉 Signal: Product becomes learning habit

---

## Phase 3 (12–24 months): Ecosystem Play

* Recruiter adoption begins
* GitHub profiles used as hiring signals
* Increase in GitHub repo activity among students

👉 Signal: Category expansion

---

# 9. 🚀 MVP Scope (V1)

### Include:

* Project recommendation
* 5–10 guided projects
* Basic AI feedback
* Project sharing

### Exclude:

* Advanced gamification
* Recruiter marketplace
* Mentor marketplace

---

# 10. ⚠️ Risks & Mitigation

---

## Risk 1: AI Over-dependence

**Mitigation:**

* Limit direct code generation
* Force hint-based assistance

---

## Risk 2: Low-quality community

**Mitigation:**

* Reputation system
* Moderation + ranking

---

## Risk 3: Feature discoverability

**Mitigation:**

* Integrate into repo creation flow
* Default onboarding for students

---

## Risk 4: Drop-offs in long projects

**Mitigation:**

* Break into micro-tasks
* Reward intermediate milestones

---
